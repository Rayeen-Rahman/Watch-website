  BUG #1 — CRITICAL  |  Buy Now Destroys Cart
File: customer/pages/ProductDetail.jsx
What goes wrong: The handleBuyNow function calls clearCart() before adding the product. If a customer had 3 items in their cart and clicks "Buy Now" on a 4th product, all 3 previous items are permanently deleted. This is a silent data loss bug.

FIND THIS CODE (around line 53 in ProductDetail.jsx):
const handleBuyNow = () => {
  // Buy Now: clear any existing cart items so only this product goes to checkout
  // This prevents leftover items from a previous 'Buy Now' session
  clearCart();
  addToCart(product, quantity);
  setIsCartOpen(false);
  navigate('/checkout');
};

REPLACE WITH (stores cart in sessionStorage, then restores after checkout):
const handleBuyNow = () => {
  // Save existing cart to sessionStorage so it can be restored after checkout
  // Only send THIS product to checkout, do not destroy other cart items
  const existingCart = JSON.parse(localStorage.getItem('watchCart') || '[]');
  sessionStorage.setItem('savedCartBeforeBuyNow', JSON.stringify(existingCart));
  clearCart();
  addToCart(product, quantity);
  setIsCartOpen(false);
  navigate('/checkout');
};

ALSO DO: In customer/pages/Success.jsx (or wherever order is confirmed), add this code to restore the cart:

  useEffect(() => {
    const saved = sessionStorage.getItem("savedCartBeforeBuyNow");
    if (saved) {
      localStorage.setItem("watchCart", saved);
      sessionStorage.removeItem("savedCartBeforeBuyNow");
    }
  }, []);

  BUG #2 — CRITICAL  |  Stale Cart Data (Price/Stock Never Refreshes)
File: customer/context/CartContext.jsx
What goes wrong: Cart items are saved in localStorage with full product objects. If you change a product price in admin from ৳2,000 to ৳1,500, a user who added it before the change will still be charged ৳2,000. Stock status is also stale — an out-of-stock product still appears in their cart as available.

FIND THIS CODE (top of CartContext.jsx, the addToCart function):
const addToCart = (product, quantity = 1) => {
  setCartItems(prev => {
    const existing = prev.find(item => item._id === product._id);
    const maxStock = product.stock ?? Infinity;
    if (existing) {
      const newQty = Math.min(existing.qty + quantity, maxStock);
      return prev.map(item =>
        item._id === product._id ? { ...item, qty: newQty } : item
      );
    }
    return [...prev, { ...product, qty: Math.min(quantity, maxStock) }];
  });
  setIsCartOpen(true);
};

REPLACE WITH (only stores id, qty, name, price, images — minimal snapshot):
const addToCart = (product, quantity = 1) => {
  setCartItems(prev => {
    const existing = prev.find(item => item._id === product._id);
    const maxStock = product.stock ?? Infinity;
    // Store only the fields needed for display + ordering
    // This keeps localStorage small and avoids deeply stale product data
    const cartItem = {
      _id:    product._id,
      name:   product.name,
      price:  product.price,
      images: product.images,
      brand:  product.brand,
      stock:  product.stock,
    };
    if (existing) {
      const newQty = Math.min(existing.qty + quantity, maxStock);
      return prev.map(item =>
        item._id === product._id ? { ...item, qty: newQty } : item
      );
    }
    return [...prev, { ...cartItem, qty: Math.min(quantity, maxStock) }];
  });
  setIsCartOpen(true);
};

  BUG #3 — HIGH  |  Empty Cart Redirects Before Success Screen Shows
File: customer/pages/Checkout.jsx
What goes wrong: After placing an order, clearCart() is called. The useEffect watching cartItems.length sees it go to 0 and immediately navigates to "/" (homepage), but only if !orderSubmitted. However there is a race condition — sometimes the state update for orderSubmitted happens a frame after the clearCart triggers, so users may get bounced to homepage instead of seeing a thank-you.

FIND THIS CODE (near line 55 in Checkout.jsx):
  // Close cart panel + redirect empty carts
  useEffect(() => {
    setIsCartOpen(false);
    if (cartItems.length === 0 && !orderSubmitted) navigate('/');
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, [cartItems.length]);

REPLACE WITH (adds a small delay to avoid the race condition):
  // Close cart panel + redirect empty carts
  useEffect(() => {
    setIsCartOpen(false);
    if (cartItems.length === 0 && !orderSubmitted) {
      // Small delay prevents race condition where clearCart triggers
      // this redirect before orderSubmitted state updates
      const timer = setTimeout(() => navigate('/'), 100);
      return () => clearTimeout(timer);
    }
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, [cartItems.length, orderSubmitted]);

  BUG #7 — MEDIUM  |  Newsletter Signup Fakes Success
File: customer/components/Footer.jsx
What goes wrong: When a user enters their email and clicks subscribe, the footer shows "You're on the list!" — but no email is ever saved. The data is lost completely. This is a trust issue with customers.

FIND THIS CODE in Footer.jsx:
  const handleSubscribe = (e) => {
    e.preventDefault();
    if (!email.trim()) return;
    // B-05 fix: was a no-op (only set state, never saved anywhere).
    // Marking as coming soon until a real email service is integrated.
    setSubscribed(true);
    setEmail('');
  };

REPLACE WITH (saves to backend and only shows success if it actually worked):
  import { API } from '../../utils/api'; // make sure this import exists at top of Footer.jsx

  const handleSubscribe = async (e) => {
    e.preventDefault();
    if (!email.trim()) return;
    try {
      const res = await fetch(`${API}/api/newsletter/subscribe`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ email: email.trim() }),
      });
      if (res.ok) {
        setSubscribed(true);
        setEmail('');
      } else {
        // Silently fail — don't alarm the user, just don't claim success
        setEmail('');
      }
    } catch {
      setEmail('');
    }
  };
BACKEND NOTE: You also need to create a POST /api/newsletter/subscribe route in your Express backend that saves the email to MongoDB. Create a simple NewsletterSubscriber model with { email, date }.

Section 3: How to Make the Website Faster on Mobile
Your Hostinger dashboard shows Mobile score of 71. Here are specific changes to push it above 85.

3.1 — Preload the Hero Image (Biggest LCP Win)
File: frontend/index.html
The hero watch image is the Largest Contentful Paint element. The browser discovers it late because it only appears in JS/React. Adding a <link rel="preload"> in the HTML head tells the browser to fetch it immediately.

FIND THIS in index.html (inside <head>):
    <link rel="icon" type="image/svg+xml" href="/favicon.svg" />

ADD THIS AFTER (add after the favicon link):
    <link rel="icon" type="image/svg+xml" href="/favicon.svg" />

    <!-- Preload hero fallback image so LCP is fast even before React loads -->
    <link rel="preload" as="image" href="/src/assets/hero_watch.png" />

3.2 — Add image compression in Vite build
File: frontend/vite.config.js
Install vite-plugin-imagemin to auto-compress PNG/JPG images at build time. This alone can reduce image sizes 50-70%.

Step 1: Run this command in the frontend folder:
npm install vite-plugin-imagemin --save-dev

Step 2: FIND THIS in vite.config.js:
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

// https://vite.dev/config/
export default defineConfig({
  plugins: [react()],

REPLACE WITH:
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import viteImagemin from 'vite-plugin-imagemin'

export default defineConfig({
  plugins: [
    react(),
    viteImagemin({
      gifsicle: { optimizationLevel: 7 },
      optipng: { optimizationLevel: 7 },
      mozjpeg: { quality: 80 },
      pngquant: { quality: [0.8, 0.9], speed: 4 },
      svgo: { plugins: [{ name: 'removeViewBox' }] },
    }),
  ],

3.3 — Use WebP images from Cloudinary (Biggest Image Win)
Your images come from Cloudinary. Cloudinary can automatically serve WebP to modern browsers. WebP is 25-35% smaller than PNG/JPG. No code change needed — just change the image URLs in your admin when uploading.

How to enable: When uploading product images, make sure you are using Cloudinary delivery URLs. Then add /f_auto,q_auto/ to the transformation string in your upload/display logic.

In customer/components/ProductCard.jsx find the resolveImg function and update it:

  export const resolveImg = (url) => {
    if (!url) return '';
    // For Cloudinary URLs, inject auto format + quality
    if (url.includes('cloudinary.com') && !url.includes('f_auto')) {
      return url.replace('/upload/', '/upload/f_auto,q_auto/');
    }
    if (url.startsWith('/uploads')) return `${API}${url}`;
    return url;
  };

3.6 — Remove the Floating Animation on Mobile
File: customer/pages/Homepage.css
The hero watch image has a continuous "float" CSS animation (translateY 0 to -12px, looping forever). On mobile this runs continuously and keeps repainting the GPU, draining battery and causing jank. It is already partially handled with prefers-reduced-motion but should also be disabled on mobile screens entirely.

ADD THIS to Homepage.css inside the @media (max-width: 768px) block:
@media (max-width: 768px) {
  /* ... existing rules ... */

  /* Stop continuous GPU animation on mobile — saves battery and reduces jank */
  .hero-image-container {
    animation: none;
  }
  .floating-spec-card {
    animation: none;
    display: none; /* already hidden at this breakpoint */
  }
}
